## ADR-001: Tech Stack

**Date:** June 2026
**Status:** DECIDED

### Context

The project needs a Python web framework, a logging approach, a configuration approach, and a set of supporting tools decided before implementation starts. The choices need to be defensible on technical grounds specific to this project, not on general framework popularity alone.

### Framework: FastAPI over Flask

Flask is minimal and well suited to small applications, and would be entirely capable of implementing the spec on its own. FastAPI was chosen instead for the following reasons.

- **Automatic OpenAPI generation and built-in validation.** FastAPI generates interactive API documentation directly from route type hints and Pydantic models, with no extra code. Request validation is built into the framework itself, invalid input is rejected before the route handler runs. Flask provides neither of these natively, both would need to be added through external libraries. Since one of this project's explicit goals is producing well documented endpoints, this is a direct fit rather than a generic framework preference.

- **Dependency injection and lifespan as a fit for layered architecture.** Two distinct mechanisms work together here:
  - Lifespan: a function that runs once when the application starts and once when it shuts down. The model is loaded during startup and kept alive for the lifetime of the application, never reloaded per request.
  - Dependency injection: a route handler declares that it needs the model service as one of its arguments, and FastAPI supplies it automatically. The handler does not construct it or know where it came from.

  Together, lifespan creates the single real model instance, and dependency injection is the channel that hands that same instance to every route that asks for it. This also enables clean testing: a fake model can be substituted through the same injection point without touching the route's code. Flask has no equivalent built-in mechanism for either of these, the pattern would have to be hand-rolled.

- **ASGI and async support, with an explicit caveat.** FastAPI is built on Starlette and is ASGI-native, meaning it can handle concurrent requests efficiently when the work being done involves waiting on I/O (network calls, disk reads, database queries). This advantage does not automatically apply to this project's core operation: the model's predict() call is CPU-bound, not I/O-bound, the CPU is actively computing for the full duration of the call, not waiting on anything external. A CPU-bound call inside an async route blocks the event loop exactly as it would in a synchronous framework. The honest justification is not "this makes prediction faster," it is that the framework supports the right pattern (offloading CPU-bound work to a thread executor) if concurrent throughput becomes a real concern later, which a synchronous-only framework does not offer as a built-in mechanism.

- **Market relevance, as a secondary factor.** FastAPI's adoption has grown substantially relative to Flask in new API projects, which is a career-relevant signal, not a technical one. Listed for completeness, does not carry weight in the technical decision above.

### Logging: structlog over the standard library logging module

The standard library's logging module can be configured to emit JSON, but only through a custom Formatter and manual handling of extra fields, with no built-in mechanism for binding context (such as a request ID) to a logger so that every subsequent log call automatically includes it. That context would otherwise have to be passed manually into every single log call across the codebase.

This matters concretely for the request ID tracing stretch goal: the intent there is that once a request ID is generated, every log line produced during that request's handling includes it without the handler code repeating it everywhere. structlog supports this directly through context binding. The decision to use structlog is therefore tied to a specific planned feature, not a general preference for one logging library's API over another.

### Configuration: pydantic-settings

Configuration (model path, log level, port) is read from environment variables through a typed Settings object rather than scattered os.getenv() calls. pydantic-settings validates these values against a schema at startup and fails immediately if a required value is missing or malformed, before the application accepts any requests. This is consistent with the project's LBYL-by-construction approach to configuration described in ADR-002.

### Testing: pytest with httpx

pytest is the standard testing framework for Python and integrates directly with FastAPI's recommended testing approach. httpx's AsyncClient allows sending requests to the FastAPI app in tests without running a real server process, exercising the actual routing and validation logic in memory.

### Package management and linting: uv and ruff

uv replaces pip, pip-tools, and virtualenv with a single tool, used here deliberately as a transferable skill, not just to get dependencies installed. ruff replaces a combination of flake8, black, and isort with a single fast tool for linting and formatting.

### Containerization: Docker and docker-compose

The service is packaged into a Docker image to guarantee it runs identically regardless of where it is deployed. docker-compose is used to run the application alongside NGINX as a separate service, rather than running uvicorn as the only process exposed to the network.

### Reverse proxy: NGINX

NGINX sits in front of the FastAPI container as the public entry point, handling TLS termination and keeping the application process from being directly reachable from outside. For a single backend service with no load balancing requirement, NGINX's routing decision is trivial, but its role as a security and operations boundary remains valuable independent of that.

### Hosting: PaaS over VPS

A PaaS that accepts Docker images directly (such as Render or Railway) was chosen over a self-managed VPS for this project specifically. A VPS would require provisioning the OS, configuring a firewall, and managing the server independently of the application, which is valuable operational knowledge but not the learning goal of this project. That knowledge is deliberately deferred to a future project where server administration is a core requirement of the problem itself, rather than absorbed here as a side effect of trying to ship on a deadline.

### Manual testing: Postman

A Postman collection covering both endpoints, with valid and invalid request cases, is maintained in the repository for manual verification during development and against the deployed service.
