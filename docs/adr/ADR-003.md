## ADR-003: Error Response Shape

**Date:** June 2026
**Status:** DECIDED

### Context

Pydantic's automatic validation errors and a manually raised HTTPException produce two different, inconsistent JSON shapes by default. The project needs one consistent shape across both, and the API should stay informative even independent of the OpenAPI docs.

### Decision

Every error response, regardless of source, is unified through a single global exception handler into:

```json
{
  "error_code": "validation_error",
  "message": "text field cannot be empty",
  "doc_url": "https://github.com/frapaparatto/py-langid-api/blob/main/docs/errors.md#validation_error"
}
```

The `doc_url` points to the per-error section on docs/errors.md. Each error uses its own anchor (`#validation_error`, `#model_unavailable`, `#prediction_failed`).

`error_code` is a machine-readable category. `message` is a human-readable reason. `doc_url` points directly to documentation about that error, so the response itself is informative even without separately consulting the docs.

A single flat `error_code` is enough for this project's scope: there is no case where multiple fields fail validation at once, since the only input is a single `text` field.

### Reasoning

Reviewed real-world precedent (Stripe, GitHub) before deciding. Both independently include a documentation link in their error objects, which directly supports the goal of the API being self-explanatory. Both also separate a machine-readable code from a human-readable message, confirming that split is the right baseline rather than a flat string.

GitHub returns a structured list of per-field errors for validation failures, since their APIs commonly have many fields that can each fail independently. That structure is unnecessary here, given the validation surface of this project, so the flatter shape was kept instead of copying it.

### Status code taxonomy

Each status code is marked by who produces it: **automatic** (FastAPI emits it without custom code), **manual** (must be implemented), or **stretch-dependent** (only occurs if a stretch feature is built).

- **400 Bad Request** (automatic): malformed or unparseable request body. FastAPI emits this before route code runs.
- **404 Not Found** (automatic): no route matches the requested path. FastAPI emits this. The global handler reshapes it into the standard error response for consistency, no routing logic is written.
- **422 Unprocessable Entity** (automatic, via Pydantic): the request is well-formed but content is invalid (missing `text`, wrong type, empty string). This is the primary error case.
- **429 Too Many Requests** (stretch-dependent): only emitted if the rate-limiting stretch feature is implemented. Until then, the API cannot produce this and it should not be documented as if it can.
- **500 Internal Server Error** (manual): an unexpected failure, e.g. the model's prediction raises. Caught by the global handler so a raw traceback never reaches the client.
- **503 Service Unavailable** (manual, conditional): the model is not loaded or not ready. Only real if a readiness check is implemented (tied to the `/health` endpoint). If not implemented, it is aspirational and should not be documented as producible.

### error_code values

The `error_code` namespace is intentionally small, matching the narrow real error space:

- `validation_error` → the 400 and 422 cases (bad or invalid input)
- `not_found` → the 404 case
- `prediction_failed` → the 500 case (model raised unexpectedly)
- `model_unavailable` → the 503 case (only if readiness check is implemented)
- `rate_limit_exceeded` → the 429 case (only if rate limiting is implemented)

### Semantic input decision: validate structure, not meaning

The API validates that input *structurally exists*, but does not judge whether the text is *meaningful enough* to classify. This follows the real-world convention for ML APIs and keeps the design coherent:

- **Empty or whitespace-only `text`** → rejected at validation, 422. This is "effectively no input," not a model judgment.
- **Non-empty but semantically thin text** (only digits, only punctuation) → accepted, passed to the model, returned with whatever confidence the model assigns. This is a model judgment, not an API rule.

### Confidence is not an error condition

A low-confidence prediction is a successful response, not an error. The API returns the predicted language and its confidence score, and the consumer applies its own business rules to decide what confidence is acceptable for its context. Baking a confidence threshold into the API would impose one judgment of "acceptable" onto every consumer, each of whom has a different context. This keeps the API a single-purpose, reusable service: predict and score, the caller decides.
