## ADR-003: Error Response Shape

**Date:** June 2026
**Status:** DECIDED

### Context

Pydantic's automatic validation errors and a manually raised HTTPException produce two different, inconsistent JSON shapes by default. The project needs one consistent shape across both, and the API should stay informative even independent of the OpenAPI docs.

### Decision

Every error response, regardless of source, is unified through a single global exception handler into:

```json
{
  "error_code": "validation_error",
  "message": "text field cannot be empty",
  "doc_url": "https://github.com/frapaparatto/py-langid-api#errors"
}
```

The URL above is illustrative, the actual target (a specific README section or docs page) will be finalized when the repo's documentation structure exists.

`error_code` is a machine-readable category. `message` is a human-readable reason. `doc_url` points directly to documentation about that error, so the response itself is informative even without separately consulting the docs.

A single flat `error_code` is enough for this project's scope: there is no case where multiple fields fail validation at once, since the only input is a single `text` field.

### Reasoning

Reviewed real-world precedent (Stripe, GitHub) before deciding. Both independently include a documentation link in their error objects, which directly supports the goal of the API being self-explanatory. Both also separate a machine-readable code from a human-readable message, confirming that split is the right baseline rather than a flat string.

GitHub returns a structured list of per-field errors for validation failures, since their APIs commonly have many fields that can each fail independently. That structure is unnecessary here, given the validation surface of this project, so the flatter shape was kept instead of copying it.
