## ADR-004: Error Propagation Strategy

**Date:** June 2026
**Status:** DECIDED


### Context

ADR-003 defines the error *response shape* (`error_code`, `message`, `doc_url`) and the taxonomy. This ADR decides a separate question: *how* an error travels from where it occurs to that response shape.

There are two coexisting patterns in FastAPI for producing an error response, and the project needs to commit to one for consistency before the error-handling code is written.


### Decision

Errors are **raised as typed domain exceptions and translated by global handlers**, not returned per-route.

- The service layer raises plain domain exceptions (`ModelUnavailableError`, `PredictionFailedError`) that carry their own `error_code`, `message`, and `doc_url`, and know nothing about HTTP.
- The route contains no `try/except`. It returns the success output (`PredictionOutput`) only, and lets exceptions propagate.
- Handlers registered once on the app (via `add_exception_handler`) catch each exception type, build an `ErrorResponse`, and wrap it in a `JSONResponse` with the correct status code.
- Pydantic validation errors are reshaped into the same `ErrorResponse` by overriding the validation exception handler.


### Alternatives considered

**Strategy A: return the error model from the route.** Each route has its own `try/except` and returns an `ErrorResponse` as a normal return value, with `response_model=Union[SuccessModel, ErrorResponse]`. Rejected for two reasons:

- A returned object defaults to **200 OK**, so an error would be sent with a success status unless extra machinery is added to set the status code.
- Error handling is duplicated across every route, and the taxonomy mapping is scattered rather than centralized.

The FastAPI docs themselves present Strategy A only to illustrate returning different model types, and note that real error handling normally uses raise-and-handle.

**Strategy B: raise and handle globally (chosen).** Wins on the points where A loses: the `JSONResponse` sets the correct status code cleanly, routes stay focused on the happy path, and the exception-type-to-taxonomy mapping lives in one place.


### Consequences

- The service stays HTTP-agnostic and reusable: it raises domain exceptions, never `HTTPException`. This also keeps it directly testable without any FastAPI machinery.
- Routes stay thin: one return path, no error branching.
- The exception-type-to-status-code mapping is centralized, and is the taxonomy from ADR-003 made concrete.
- Because handlers return a `JSONResponse`, the automatic `response_model` step does not run on them, so each handler constructs `ErrorResponse` manually to guarantee the shape. This is the deliberate cost of controlling the status code.
- The route's `response_model` declares only the success model (`PredictionOutput`), not a `Union` with `ErrorResponse`, since errors never flow through the route's return value. Declaring the error model there would document a path that does not happen.


### Where the taxonomy lives: on the exception, one handler

Each domain exception carries its **full** taxonomy entry as attributes: `status_code`, `error_code`, `message`, and `doc_url`. The status code lives on the exception as a class attribute, not in the handler.

This allows a **single handler registered on the base** `LanguagePredictionError` to serve every domain error. By handler resolution through inheritance, a handler registered for a base class catches all its subclasses unless a more specific handler is also registered. Since every domain error builds the same response shape and differs only in its values, one base handler reads the four attributes off whatever exception was raised and builds the response. No per-type handler and no per-type branching is needed.

**Rationale and tradeoff:**

- One handler instead of one per exception type removes duplication. Adding a new error type later is just a new subclass with its four values, no new handler.
- It also sidesteps a type-checking issue: a handler typed against the base exception matches the expected handler signature, whereas a handler typed against a specific subtype trips a strict checker on parameter contravariance.
- The tradeoff: a single handler cannot do different logic per type (for example extra logging for one type only). If a specific type ever needs special handling, a more specific handler can be registered for just that type, and the base handler still catches the rest. For a project where all domain errors are handled identically except for their values, the single base handler is the cleaner choice.

A separate detail: the base exception calls `super().__init__(message)` so it behaves like a proper Python exception (the message is stored in `args`, so `str(exc)`, tracebacks, and logs show a meaningful message). This serves the Python-level representation, separate from the custom attributes that serve the HTTP response.


### The 404 case: registered on the status code, not an exception

`not_found_handler` is registered directly on the integer status code `404`, not on a `LanguagePredictionError` subtype. FastAPI raises a plain `HTTPException(status_code=404)` itself when no route matches, before any route or service code runs, so there is no domain exception to key a handler off. FastAPI's `exception_handlers` mapping accepts either an exception type or an int status code as a key; `404` is the one place this project uses the int form. The handler returns a fixed message rather than reading `exc.detail`, since FastAPI's default detail ("Not Found") does not follow this project's lowercase, no-punctuation message style.
