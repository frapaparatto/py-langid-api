## ADR-002: Error Handling Strategy

**Date:** June 2026
**Status:** DECIDED

### Context

The project needs a consistent error handling strategy across the domain, service, and HTTP boundary layers before implementation starts, since retrofitting consistency after the fact is more expensive than deciding it up front. Python has no equivalent to std::optional, and offers two coexisting idioms (LBYL and EAFP) without a single canonical rule for when each applies. The strategy must also cover where Pydantic validation fits relative to manual error handling.

### Decision

Four error handling mechanisms, each with a specific role:

- **assert for programmer errors.** Preconditions that should never be violated if the code is correct. Removed when Python runs with the `-O` flag, same principle as C's NDEBUG. The practical test: if this check fails, does it mean the developer made a mistake, not the caller or the environment? If yes, assert.
- **Exceptions for rare runtime errors.** I/O failures, model loading failures, and any condition where recovery is possible but the program cannot proceed normally. `raise ValueError` for validation and constraint errors. `raise RuntimeError` for infrastructure and I/O errors where no more specific built-in exception fits. Specific built-in exceptions (`FileNotFoundError`, `KeyError`) are used whenever they match precisely rather than falling back to a generic type.
- **Returning None for expected absence.** A condition is not a failure if it is a normal, anticipated outcome of the operation, such as a lookup that legitimately finds nothing. The caller checks for `None` explicitly before using the result.
- **Pydantic validation at the HTTP boundary.** Any data arriving from outside the system (the request body) is validated by Pydantic before the route handler function is ever called. The handler never receives malformed input by construction. Validation errors become a 422 response automatically, without the handler writing any validation code itself.

### Choosing between LBYL and EAFP

The dividing line is not the type of data, it is whether checking a condition and acting on it can be cleanly separated into two steps without a race condition or duplicated work.

- **LBYL (Look Before You Leap):** used for expected absence and for any check where verifying the condition first does not change the cost or correctness of the subsequent action. A dictionary lookup checked with `if key in dict` before access is a clean LBYL case, the check and the access do not interfere with each other.
- **EAFP (Easier to Ask Forgiveness than Permission):** used for I/O and any operation where the check and the action cannot be meaningfully separated, because the underlying condition can change between the check and the action, or because checking in advance duplicates work the operation will do anyway. Loading the pickled model file is the clearest instance of this in the project: checking that the file exists and is readable, then opening it, is not actually safer than attempting to open it directly and catching the failure, since the file could disappear or become unreadable in the gap between the two steps.

**Boundary validation is a third category, distinct from both.** Pydantic enforces the shape of incoming HTTP requests before the handler function exists in the request's path. This is not the developer choosing to check before acting, it is the framework rejecting invalid input upstream, by construction, before there is any handler code to write either style against.

### Practical mapping for this project

ModelService loading the pickle file at startup: EAFP, wrapped in try/except, since failure here is an infrastructure condition and the check-then-open sequence would not be more reliable than attempting the load directly.

ModelService.predict() called with already-validated text: no defensive check needed before the call, since Pydantic has already guaranteed the input shape. Any exception the model itself raises during prediction is caught by the global exception handler, not by ad hoc validation.

Configuration loaded via pydantic-settings: LBYL by construction, since pydantic-settings validates environment variables against the Settings schema at startup and raises immediately if a required value is missing or malformed, before the application is allowed to start serving requests.

### Exception message format

Messages are lowercase, no "Error:" prefix, no exclamation marks. The exception's message is a clean, specific reason: "model file not found at configured path", "text field cannot be empty". The global exception handler at the application boundary is responsible for converting any uncaught exception into a structured JSON error response, the exception itself stays a plain, readable string.

### Never let a raw exception reach the client

Any unhandled exception is caught by a global exception handler registered in main.py and converted into a structured JSON error response. A raw Python traceback must never reach an HTTP response.
