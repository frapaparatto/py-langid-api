## ADR-006: Testing Strategy

**Date:** July 2026
**Status:** DECIDED


### Context

The API works, but working code is only half of a dependable application: its correctness has to be verifiable, and stay verifiable as the code changes. Automated tests are the safety net that makes change safe. They let the endpoint logic be refactored, dependencies updated, or the model swapped, with confidence that existing behavior has not silently broken.

This ADR records not a list of tests, but the **judgment** behind them: what is worth testing, what is not, where the boundaries between kinds of tests lie, and why. The skill this codifies is deciding *what deserves a test*, which matters more than the mechanics of writing one.


### The principles that guide what to test

**Test behavior, not implementation.** A test should assert what the code *does* (its contract: this input produces this output), not *how* it does it. The check: if a function is refactored so its shape changes but its behavior does not, a good test still passes. A test that breaks on a pure refactor was testing the implementation, and is a liability, it creates false alarms and discourages refactoring. Only tests that break when behavior actually changes are worth keeping.

**Allocate test effort by risk.** Testing is not uniform coverage; it is spending effort where a bug would be both plausible and costly. The prediction endpoint's error paths are the app's contract with its clients, high risk, worth testing thoroughly. A trivial, linear piece of logic with no real failure mode is low risk and needs little or nothing. Risk-based does not mean skipping low-risk components entirely: it means testing the part *of* them that carries risk. The health endpoint mostly does nothing interesting, but the one behavior that could break, that it reports the model's real readiness state, is worth a test; the fact that it returns 200 is not.

**Fake the dependency, never the response.** This is the distinction that governs the whole suite. The response is what a test asserts on, it is the output under test, so faking it would test nothing (asserting a fake equals a fake). What gets substituted is the *dependency* (the model), the input to the pipeline. The real pipeline then produces a real response, and the test asserts on that real response. Substituting the model makes the test deterministic and independent of a real model file, while still exercising validation, routing, and response shaping for real.

**Mock to trigger failure modes, not to hide the risk.** A lesson carried from an earlier project: mocking the exact thing that carries the risk means testing nothing. If the real danger is malformed data at a serialization boundary, mocking that boundary away with always-clean fake data tests only the happy path and never the real failure. So faking is used for two legitimate reasons, and never to fake away the thing under test:
- to remove a dependency as a variable so the surrounding pipeline can be tested deterministically (a fake model returning a known good value), and
- to *produce* a failure mode that is hard to trigger with the real thing (a fake model that raises, so the error-wrapping branch can be tested).


### Unit versus integration: where the boundary sits

The two kinds of test differ in what is real and what is faked, and each targets a different risk.

**Unit tests** isolate a single unit and fake what it depends on. They are the right tool for the **branches**: the code paths a real model does not easily exercise. The interesting behavior of the prediction logic is not its linear happy path (trivial, low-value to test), but its error branches, that it raises the model-unavailable error when the model is absent, and wraps an unexpected failure into the prediction-failed error when the model raises. Testing these needs a fake that misbehaves on command (an absent model, or one that raises), which is exactly what a unit fake is for.

**Integration tests** exercise the real request-response flow through the app, faking only what must be controlled for determinism. They are the right tool for the **pipeline seam**: does a request validate correctly, route to the prediction path, invoke the model, and come back in the right shape. Here the model is faked (via the injection seam) to a known value so the test is deterministic, but everything around it, validation, routing, error handling, response shaping, is real. These tests do not evaluate the model's statistical accuracy; that is a separate concern belonging to the model's own development. They verify that the pipeline *surrounding* the model works: that the API calls the prediction path and transforms the result correctly.

The injectable-model design (the model provided through a dependency rather than reached for directly) is what makes this clean: the dependency is the seam a test substitutes a fake through, without touching the route code. Testability was one of the reasons that design was chosen; here it pays off.


### What the suite covers, and why each earns its place

**The prediction endpoint** is the core contract, so it is tested across its outcomes: the happy path (valid input, a fake model returning a good value, asserting the successful response and its shape), and each error outcome (an absent model producing the 503 shape, a raising model producing the 500 shape). In all of these the model is faked and the *real response shape* is asserted, the pipeline is what is under test, not the model.

**Validation errors** are the simplest and cleanest test in the suite, and understanding *why* is itself the point. Invalid input is rejected by the schema layer *before* the route runs, before the model is ever reached. So a validation test needs neither a real model nor a fake one: the request never gets far enough to touch it. Send invalid input, assert the 422 and the unified error shape. The confusion of "should the model be loaded or faked for this test" dissolves once it is clear the input never passes the boundary to reach the model.

**The health endpoint** gets one behavior tested, in two states: that it reports the model's real readiness (loaded versus not) in the response body, always with a 200. The model state is controlled and the body asserted. Its uninteresting parts (that it returns 200 at all) are not worth separate tests.

**Model loading (the startup lifecycle)** has a real failure mode worth covering: that a present model file results in a loaded model, and a missing one leaves the model unset (which is what feeds the unavailable-error path at request time). This is more involved to drive in a test than the endpoint cases and depends on mechanisms not yet learned, so it is planned but deferred until the setup pattern for it is understood, rather than specified prematurely.

**The middleware and logging are deliberately not given dedicated tests.** Logging is observability, a monitoring concern, not behavior a client depends on, and testing log *content* is brittle (it breaks when wording changes) for little value. The one middleware behavior that genuinely matters, that an exception passing through is re-raised rather than swallowed, is not tested directly because it is already exercised *transitively*: if the middleware swallowed exceptions, the prediction error-path tests would fail, because they would never receive the error response. So the important behavior is covered by the error-path tests, and the observability behavior is not worth the brittleness of asserting on it. This is a proportionality decision: the request-id generation is trivial and low-risk, and the re-raise is covered elsewhere.


### The tooling

Tests run against the app in-process, with no server. FastAPI's `TestClient` (built on `httpx2`) sends real requests directly to the app object and returns the real responses, so the tests are fast and self-contained. The client is stateless between calls, so it is built once (as a fixture) and shared across the tests in a file.

The dependency-injection system provides `dependency_overrides`, the seam through which a fake model is substituted for the real one in a test, without touching the route code. This is the concrete payoff of injecting the model through a dependency: the same seam that decouples the route from the model at runtime is what a test swaps a fake through.

Each mechanism is used because a test identified above needs it, `TestClient` to exercise the request-response flow, `dependency_overrides` to control the model, not because the tooling suggests a shape for the suite.


### Consequences

- A small, focused suite: the prediction endpoint across its outcomes, validation at the boundary, the health readiness behavior, and (deferred) model loading. Roughly the behaviors that carry real risk, and no more.
- No dedicated logging or middleware tests; the middleware's one important behavior is covered transitively by the error-path tests.
- Tests assert on real responses produced by the real pipeline, with only the model faked, so they survive refactors that keep behavior intact.
- The injectable-model design is validated by how cleanly it supports substitution in tests.
