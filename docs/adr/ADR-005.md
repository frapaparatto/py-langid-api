## ADR-005: Logging Strategy

**Date:** June 2026
**Status:** DECIDED


### Context

The API needs observability: a record of what happens while it runs, so failures and behavior can be understood after the fact, especially in production where events cannot be watched live. Milestone 2 used `print()` for startup messages, which has no levels, no structure, and no control over output. This ADR decides what to log, at what severity, in what format, and where the logging lives, before the logging code is written.

The relevant question is not only *how* to log (the library) but *what* deserves logging and at what level, which is a design decision, not a library detail.


### Decision

Use **`structlog`** for **structured logging**: each log entry is a set of key-value fields (an event dict), rendered as JSON in production and as human-readable console output in development. Structured logs are queryable by tools; unstructured strings are not. `structlog` makes structured events the natural default, whereas the standard library is string-first and would require custom formatters to reach the same result.

**Configuration is centralized; log calls are distributed.** Logging is configured once at startup (in `app/core/logging.py`): the processor chain, the renderer, the level, the destination. Individual log calls happen at the site of each event (lifespan, middleware, handlers), using a logger that obeys the central config. A log call belongs where the event happens so it has natural access to the relevant context (the exception, the request, the path); routing it through one central function would strip that context.


### The event taxonomy

Three categories of events are logged, appropriate for a small single-purpose API: **lifecycle**, **request/response**, and **errors**. The prediction itself is not logged as an event; the request/response boundary around it is, which is the correct granularity.

| Event | Level | Where | Context |
|---|---|---|---|
| Model loaded | INFO | lifespan | path |
| Model failed to load | ERROR | lifespan | path, error cause |
| Startup / shutdown complete | INFO | lifespan | |
| Request arrived | DEBUG | middleware | method, path, request id |
| Request completed | INFO | middleware | method, path, status, duration, request id |
| Validation error (422) | WARNING | handler | request id, validation message |
| model_unavailable (503) | ERROR | handler | request id |
| prediction_failed (500) | ERROR | handler | request id, error cause |


### Two principles behind the level assignments

**Severity is contextual, not a property of the event type.** "Model failed to load" is an ERROR rather than a WARNING because the model is this app's core capability: the process survives, but a core function is dead. The same event in an app where the model was optional might only be a WARNING. The level reflects what the event means for *this* app.

**Client errors (4xx) versus server errors (5xx).** A client sending invalid input (422) is expected operation, the app behaved correctly by rejecting it, so it is a WARNING, not an ERROR. ERROR is reserved for server-side failures that are the app's own problem: prediction_failed (500) and the model-unavailable (503) condition. This maps directly onto the error taxonomy from ADR-003.


### Request logging: both boundaries, at different levels

Request arrival is logged at DEBUG and request completion at INFO. In production (INFO threshold) only completion is emitted; while debugging (DEBUG threshold) both are. This preserves the ability to detect requests that arrived but never completed (hung or crashed mid-processing) without doubling log volume in production.


### Format and level: two independent switches

Log level and output format are configured separately, via two settings added to the existing pydantic-settings Settings class:

- LOG_LEVEL (default INFO): the threshold; entries at or above it are emitted.
- LOG_FORMAT (default json): the renderer, json or console.

Both are defaulted and independently overridable. In development, LOG_LEVEL=DEBUG and LOG_FORMAT=console are set in .env; in production the defaults (or platform-injected env vars) give INFO and json. There is no branching logic: the settings are the switches, read once at startup when logging is configured.

Two explicit settings are preferred over a single DEBUG flag that toggles both, so each setting maps to exactly one behavior with no hidden coupling, and so the logging configuration composes with the existing environment-based config (ADR on configuration / pydantic-settings).


### Destination

Logs are written as structured JSON, per the project requirement to log to a file for later analysis. Console output is the development alternative, more immediate while working. A common modern alternative (not chosen here) is to write JSON to stdout and let the platform capture it; the file destination is a project-specific choice.


### Consequences

- The `print()` calls in `main.py` are replaced by structured log calls.
- Startup and shutdown events are logged in the lifespan; per-request events (with a request id bound for the duration of the request) are logged in middleware; error events are logged in the exception handlers where they are caught.
- The request id, bound once per request, correlates every log line produced during that request.
- The `raise ... from e` chaining already present in the service means the underlying cause of a prediction_failed is available to be logged as structured data.
- Never log secrets, credentials, or full request bodies containing user data; logs are stored and shipped to tools, so a logged secret is a leaked secret.
