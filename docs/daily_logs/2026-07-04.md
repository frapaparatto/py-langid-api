## 2026-07-04

### What I built
- Implemented structured logging with structlog: app/core/logging.py's setup_logging() configures the processor chain (contextvars merge, log level, ISO timestamp, exc_info) and switches between a JSON renderer and a console renderer based on LOG_FORMAT.
- Added log_level and log_format to the Settings class (config.py) as Literal types, so an invalid value (a typo like "INFOO") fails validation at startup instead of silently misbehaving.
- Replaced the print() calls in main.py's lifespan with structured log calls: startup, model_loading, model_loaded, model_load_failed (with log.exception for the unexpected-error branch), shutdown.
- Added an HTTP middleware in main.py that binds a request_id via structlog's contextvars for the duration of each request, logs request_started at DEBUG and request_completed at INFO with duration_ms, and request_failed with a full traceback if the route raises.
- Updated the exception handlers (handlers.py) to log: validation errors at WARNING (client mistake, 4xx), and domain errors at ERROR, using log.exception for PredictionFailedError specifically so its chained cause (the underlying sklearn exception) is captured.
- Wired LOG_LEVEL and LOG_FORMAT into docker-compose.yml and .env.example.
- Updated ADR-005 to reflect the actual destination decision: stdout instead of a file, per the 12-factor principle.

### What I learned
- Structured vs unstructured logging: a log line as a human sentence is only regex-parseable by a machine; a log entry as key-value pairs is queryable and filterable by field. structlog is data-first (the entry starts as a dict, rendering to text is the last step), the standard library is string-first.
- Severity is a property of what an event means for this specific app, not a property of the event type. "Model failed to load" is ERROR here because the model is the app's core capability, not because failed-load events are inherently ERROR everywhere.
- The client (4xx) vs server (5xx) distinction maps directly onto WARNING vs ERROR: a validation error is the app behaving correctly by rejecting bad input, so it does not deserve the same severity as a genuine server-side failure.
- Context binding (structlog.contextvars) solves the concurrent-request correlation problem: bind a request_id once per request, and every log line emitted during that request carries it automatically, without threading it through every log call by hand.
- 12-factor logging: the app's job is to emit a structured event stream, not to manage where it ends up. Writing to a file couples the app to filesystem concerns (rotation, disk space, paths) that belong to the environment, not the app.
